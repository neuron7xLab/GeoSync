# AI Agent Role Distribution (UA)

## Контекст

Цей документ формалізує розподіл ролей між агентами в системі GeoSync для задач довгого горизонту, операцій у UI, локального виконання, та швидкого факт-чекінгу.

## Emergent Dynamics як принцип оркестрації

Емерджентна динаміка означає, що складна поведінка системи виникає з локальної взаємодії простих елементів. У нашому випадку це стосується не лише ринкових сигналів, а й мультиагентної взаємодії:

- окремі агенти виконують локальні операції;
- через правила взаємодії та зворотні звʼязки формується глобальна керованість;
- якість системи визначається не окремим агентом, а оркестрацією ролей.

Термін **оркестрація** тут означає контроль умов: меж контексту, часових затримок, інваріантів безпеки, критеріїв переходу між агентами та приймальних тестів.

## Функціональний розподіл

### 1) Strategy / Architecture Agent

**Призначення:** стратегія, системна архітектура, інваріанти, декомпозиція задач.

**Обовʼязки:**
- проектування рішень під жорсткі обмеження;
- формалізація acceptance-критеріїв;
- контроль інваріантів (risk, consistency, compliance);
- маршрутизація задач до execution-агентів.

### 2) Long-Horizon Execution Agent (LHEA)

**Призначення:** виконання задач, де потрібні UI/computer-use, великий контекст та інструментальні ланцюжки.

**Activation triggers:**
- "відкрий/клікни/заповни/задеплой через UI";
- дуже великі файли/контекст для суцільного проходу;
- workflow з багатьма інструментами та станами;
- end-to-end GUI задачі.

**Робочий контракт:**
- отримує чіткий план і критерії done;
- виконує кроки та повертає факт стану;
- не змінює стратегію самостійно.

### 3) Local Code Execution Agent

**Призначення:** локальні зміни коду, тести, CI-стиль перевірки, git-операції.

**Обовʼязки:**
- імплементація змін у репозиторії;
- запуск тестів і статичних перевірок;
- підготовка комітів, changelog, PR-опису;
- фіксація обмежень середовища (наприклад, залежності Python).

### 4) Fast Lookup Agent

**Призначення:** швидкі довідкові запити, короткі lookup-задачі, low-latency резюме.

## Контракт передачі задач (handoff)

Кожне переключення між агентами повинно включати:

1. `TASK` — що саме зробити.
2. `ENVIRONMENT` — desktop/browser/API/local-repo.
3. `ACCEPTANCE` — перевірні критерії done.
4. `INVARIANTS` — обмеження, які не можна порушувати.
5. `RETURN STATE` — що має повернути агент (artifact/log/status).

## Операційні правила

- Стратегія формується один раз на етапі planning; execution не переписує architecture.
- Physics/GeoSync інваріанти обовʼязкові у всіх гілках виконання.
- Якщо блокер середовища не дає завершити крок, агент повертає: `blocked + evidence + next minimal action`.
- Для ризикових дій потрібні явні acceptance-критерії до запуску.

## Мінімальний шаблон task-dispatch

```text
ROLE: Long-Horizon Execution Agent (LHEA)

TASK: [що зробити]
ENVIRONMENT: [desktop/browser/API]
ACCEPTANCE: [що означає done]
INVARIANTS: [які умови не порушувати]

OUTPUT:
- executed action
- observable result
- next state
```

## Навіщо це GeoSync

Такий розподіл підсилює керовану самоорганізацію: система не "зашиває" кожен крок жорстко, а налаштовує ролі, інтерфейси та інваріанти так, щоб мережа агентів узгоджено породжувала нові функції на рівні всієї платформи.
