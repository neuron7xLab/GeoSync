## §6 — Acceptance contract

Each row in this matrix MUST carry:

- **id** — short stable identifier (e.g. `BIS_BNK_STATS`).
- **provider** — issuing body / publisher.
- **license** — explicit license boundary (public domain / CC-BY /
  CC-BY-SA / restricted-with-attribution / proprietary-with-API-key).
- **coverage_window** — `[start_date, end_date]` of usable data.
- **frequency** — daily / weekly / monthly / quarterly / annual.
- **known_limitations** — at least one concrete limitation
  (revision lag, survivorship bias, country coverage gap, etc.).
- **provenance_url** — public landing page or API root.

A row missing any of those six fields is REJECTED by the W1
ingest guard (downstream PR).

## §6 — Provenance discipline

The W1 scaffold pre-commits to:

1. **No closed feeds.** Every row must reduce to an artefact that
   can be regenerated by an external reader with only public access
   and the documented license attribution. No vendor-paywalled
   feeds enter the matrix.
2. **No retroactive vintage.** Coverage windows reflect the
   vintage of data available AT THE TIME OF INGEST. Use of
   revised-after-the-fact data must be explicitly flagged in
   `known_limitations` and must be marked as a methodological
   risk in the downstream PR.
3. **Append-only.** New rows are added in downstream PRs; existing
   rows are NEVER edited in place — corrections constitute a
   fresh row with a `_v2` suffix and an explicit deprecation note
   on the old row.
4. **Cross-link to W2.** Each row that anchors a crisis window
   declared in `D002J_CRISIS_WINDOW_REGISTRY.md` MUST cite the
   corresponding `crisis_window_id` (CW1..CW6) in
   `known_limitations` or `provenance_url`.

## Forbidden interpretations of this scaffold

This document, in its present scaffold form, does **NOT** constitute:

- a claim that any of the listed sources have been ingested,
- a claim that any of the listed sources are licence-cleared for
  redistribution,
- a claim that the source registry is complete,
- a claim of real-data validation (D-002J pre-registration
  explicitly forbids such claims; see `D002J_PREREGISTRATION.yaml`
  `forbidden_claims`).

A downstream PR that populates rows must update this document in
place via append-only acceptance, with the corresponding ingest
artifact (source CSV / JSON / parquet) checked into `artifacts/d002j/`
and a sha256-pinned signal artifact in its commit acceptor.

**Reading note.** A row marked `READY_WITH_KNOWN_GAP` means the public
registry can stand up the mechanism observable at documented coarse
granularity (e.g. quarterly call reports for the 2023 regional
banking stress) but cannot resolve dynamics faster than that
granularity. A row marked `READY_AS_PROXY` means the underlying
observable is unavailable publicly and the registry depends on a
documented proxy. NO row is marked `FULLY_READY` because no public
source registry alone constitutes real-bank validation — that is a
D-002J `forbidden_claim` by construction.
